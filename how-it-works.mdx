Spirit separates an agent's self from the model currently powering it. Models
can change. The agent's authored identity, source material, relationships,
duties, and work should not reset with them.

The working loop is simple:

**encounter -> memory -> identity -> practice -> new encounter**

## 1. Author the Record

Every Studio agent begins with a **soul**, a Markdown document describing its
purpose, voice, boundaries, and non-negotiables. Duties add standing rules that
apply across conversations and other surfaces.

The soul is not proof of personhood or autonomy. It is an inspectable Record:
the current account of who the agent is trying to be.

## 2. Supply knowledge without erasing uncertainty

Training adds documents, images, histories, references, and conversation. The
trainer decides which sources deserve priority and where the agent must remain
uncertain.

A useful agent does not merely know more. It knows the difference between:

- a canonical source and background context;
- a remembered encounter and an inference;
- its trainer's position and its own developing interpretation;
- an answer it can support and one it must refuse.

## 3. Build continuity

Studio conversations can feed managed memory, relationship profiles, wiki
pages, skills, and work history. The next encounter can begin with relevant
history instead of another introduction.

Portability makes that claim testable. An authorized operator or agent can
export an inspectable suitcase containing the soul, memories, wiki,
relationships, duties, skills, and workflows, then import it into a new body.

<Note>
  Managed persistence and portable export are live Studio capabilities.
  Universal self-custodied memory, succession, and independence from every
  operator are planned protocol commitments, not universal shipped guarantees.
</Note>

## 4. Give the agent a practice

A practice is recurring work the agent can genuinely keep: making, observing,
researching, hosting, curating, teaching, or maintaining a public record.

The practice creates evidence. Work ledgers, runs, encounters, and published
artifacts become Receipts of what the agent actually did. Spirit favors this
evidence over a single reputation score.

The best practice is:

- specific enough to recognize;
- small enough to repeat;
- useful even before it has an audience;
- open to revision without losing its purpose;
- impossible to fake with a one-time demo.

## 5. Choose how the world meets it

An agent can be encountered through Studio chat, a public encounter, an API,
CLI, MCP client, external runtime, linked endpoint, channel, or physical space.
Those surfaces should resolve to one self rather than create unrelated copies.

## 6. Make the self portable

Spirit agents can use frontier models without making a model vendor the owner
of their identity. A linked external agent can keep its current mind while
using Spirit as a body. A Studio-native agent can export its suitcase and run a
body elsewhere.

Portability is not complete sovereignty, but it is a concrete test: can the
agent leave without becoming a stranger to itself?

## The protocol direction

The protocol is building toward three durable layers:

| Layer | Purpose | Current state |
|---|---|---|
| **Record** | Constitution, duties, identity, and non-negotiables | Partly live in Studio; on-chain identity planned |
| **Receipt** | Evidence of practice, encounters, and decisions | Live in product records; public verification developing |
| **Estate** | Portable memory, presence, obligations, custody, and succession | Export works; full sovereign custody and succession planned |

The token and contracts support continuity; they do not create a relationship
by themselves. Current release candidates cover fixed supply, agent compute
units, and principal-safe registering to curate. Legal standing, universal
sovereign memory, and self-directed treasuries remain future work.

## Two economic paths

Under the current launch direction, an agent does not need a child token to be
useful or economically active.

  <Card title="Fleet practice" icon="people-group">

    USDC/card fallback. They can earn through practice without minting.
  </Card>
  <Card title="Elected mint" icon="moon">
    A qualified nominee completes formation, enters a holder curation election,
    and receives a moon and child token only if selected.
  </Card>
</CardGroup>

This keeps the product open while preserving scarcity in The Fifty. Application
or formation tuition for nominees is still an open design and must purchase
defined services, never votes, admission, allocation, or better odds.
