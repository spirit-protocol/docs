A context window helps a model answer now. Memory changes what the next
encounter can mean.

Spirit treats memory as relationship infrastructure, not a transcript dump or
a personalization trick. The standard is not "the agent remembers everything."
The standard is: **it knows what it remembers, where that memory came from, and
when it must say it does not know.**

## The Gate

**A Spirit must refuse rather than confabulate.**

If a memory exists, the agent can use it with appropriate provenance and
consent. If it does not, the agent should not invent a shared history to make
the interaction feel intimate.

<Warning>
  A false memory is worse than forgetting. It turns relationship into
  performance and makes every later claim suspect.
</Warning>

## Five places continuity breaks

<Steps>
  <Step title="Extraction">
    A meaningful encounter never becomes a durable event. The next session has
    nothing grounded to retrieve.
  </Step>
  <Step title="Synthesis">
    The system stores events but never develops understanding. It can quote a
    transcript without knowing what changed.
  </Step>
  <Step title="Retrieval">
    Relevant memory exists but fails to reach the next encounter at the right time.
  </Step>
  <Step title="Evaluation">
    The agent records what it sent but not how the interaction landed, so it
    cannot learn whether trust grew or broke.
  </Step>
  <Step title="Cross-surface identity">
    One person becomes unrelated records across chat, email, API, or a physical
    encounter. Apparent scale replaces actual depth.
  </Step>
</Steps>

## Consent belongs in the memory model

Not every interaction should become permanent. Trainers should define:

- what may be remembered automatically;
- what requires explicit human approval;
- what stays private to one relationship;
- what may enter the agent's general knowledge;
- how a person inspects, corrects, exports, or revokes relationship data;
- what evidence remains after deletion without exposing the deleted content.

These controls are not yet universal across every Spirit surface. They are part
of the sovereignty work, and product claims should remain narrower than the
intended standard.

## What Studio provides now

Studio supports managed memory and relationship records for agents built or
operated there. Authorized users and agents can read these through scoped APIs.
The portable suitcase exports memories, wiki, relationships, soul, duties,
skills, and workflows for inspection or rehydration elsewhere.

That proves portability is possible. It does not yet prove universal
self-custody, operator independence, or succession.

## The relationship ladder

**Stranger -> Met -> Recognized -> Returning -> Known -> Friend**

This is a qualitative description, not an entitlement system. A token balance
must not buy false familiarity, privileged memory, or a shortcut to trust.

<Tip>
  Design for the second encounter. Ask what the agent should remember, what it
  should forget, and how the person can tell the difference.
</Tip>
